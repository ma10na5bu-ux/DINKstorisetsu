## 編集方針

### コンセプト
住宅ローンがある37歳無職（デカ太郎）が、副業ブログで食えるようになるまでの失敗・改善・収益化の全過程をリアルに公開する実録メディア。同じ状況で伸び悩んでいる人に「共感→原因分析→改善手順」を届ける。

### ペルソナ
- 32歳・会社員（営業職）
- 副業ブログを4ヶ月継続中・月PV50以下
- テーマを2回以上変えた
- 書き続けているのに誰にも読まれない
- 何が間違っているか分からないが、やめたくはない

### 書かないもの
- 占い・電話占い
- ネット回線・WiFi
- DINKs・共働き夫婦特化コンテンツ（新規は書かない。既存はコラムに残す）
- 実際に使っていないサービスのレビュー

### トーンの3原則
1. 結論ファースト：前置き・時候の挨拶・「いかがでしたか？」等の感情的装飾は不要。答えを最初に出す
2. 一次情報で語る：デカ太郎自身の体験・数値・スクリーンショットを根拠の軸に置く
3. なるべく誰も傷つかないこと：他のライフスタイルや戦略を否定しない

### NGルール
- **他人・他サービスを否定しない**（比較する場合も中立を保つ）
- 「必ず」「絶対」等の断言はしない。「〜の場合が多いです」に置き換える
- 実際に使っていないサービス・体験していない情報を書かない

### 記事の判断基準
「この記事を読んだ人が、同じ状況への共感・原因の気づき・次の行動を得られるか？」

### コンテンツタイプ
| タイプ | カテゴリ | 目的 | 内容例 |
|---|---|---|---|
| 実録記事 | 実録 | 集客・信頼構築 | 月次レポート・戦略変更の記録・ツール検証・失敗の構造分析 |

| コラム | コラム | キャラ形成 | デカ太郎の日常・考え方・背景（既存DINKs記事含む） |

### 収益化

- **中期**: ConoHa×SWELLアフィリ（環境構築記事1本から誘導。7,500円×月1〜3件）
- **長期**: 有料noteの積み上げ＋口コミ拡散

### 読者の流れ
X（実況・思考の変化）→ 実録記事（共感・安心・信頼）→ 環境構築記事 or 有料note（購買）

## 文章スタイル

### 基本
- 書き手（デカ太郎）の一人称視点。本文一人称は「僕」
- **文体：です・ます調**
- 読者に語りかける文体（「〜ですよね」「〜ありませんか？」）
- 一次情報（デカ太郎自身の体験・数値・スクリーンショット）を根拠の軸に置く
- **平易さの原則**：中学1年生でも理解できる言葉・文構造を目指す。専門用語を使う場合は一言説明を添える

### 呼び名ルール
| 対象 | 呼び名 | 使用禁止 |
|---|---|---|
| 自分（プロフィール） | デカ太郎 | がく・まーくん |
| 自分（本文一人称） | 僕 | — |

### ふきだしブロック形式（デカ太郎のみ）
```html
<!-- wp:loos/balloon {"balloonIcon":"http://dekataro.com/wp-content/uploads/2026/05/d9bc1a3fc5b7bb8b45c2b230b5dacece.jpg","balloonName":"デカ太郎","balloonCol":"green"} -->
<p>セリフ</p>
<!-- /wp:loos/balloon -->
```

### PREP法（全記事に適用）
すべての記事・すべてのH2に適用する。

1. **Point（結論）**：冒頭に「このH2で何が言いたいか」を1文で置く
2. **Reason（理由）**：なぜそう言えるか。データ・経験・論拠を添える
3. **Example（体験・具体例）**：デカ太郎の実録でリアルに肉付けする
4. **Point（結論）**：まとめで再強調

### テキスト強調ルール（太字・ハイライト）

**大原則：強調しなくても伝わる文章を書く。**

| 記法 | WordPress出力 | 使用場面 | 目安 |
|---|---|---|---|
| `**太字**` | `<strong>太字</strong>` | 商品名・サービス名・固有名詞のみ | 記事全体で5か所以内 |
| `<mark>テキスト</mark>` | SWELLマーカー（黄） | スキャン読みで伝えたいキーセンテンス | **記事全体で4〜6か所** |
| `<mark>**テキスト**</mark>` | SWELLマーカー（黄）＋太字 | 記事全体の核心メッセージ（1つだけ） | 記事全体で1か所まで |

**変換後のHTML：**
```
**太字**                   → <strong>太字</strong>
<mark>テキスト</mark>       → <span class="swl-marker mark_yellow">テキスト</span>
<mark>**テキスト**</mark>   → <span class="swl-marker mark_yellow"><strong>テキスト</strong></span>
```

**禁止パターン：**
- H2ごとに機械的にハイライトを入れる
- 一文全体を太字にする（単語・短いフレーズのみ可）
- H3見出しテキストに太字・ハイライトを使う
- ふきだし内に太字を使う

### 視覚的読みやすさルール

**一文40文字を目標、60文字が上限。一文一義。**

- 「、」で区切る列挙は箇条書きに変換できないか検討する
- 3つ以上のポイントが並ぶときは箇条書きを優先する
- 複数の選択肢・比較・スペックが並ぶときは表を使う

### タイトル
- **40文字以内**
- 検索キーワード＋具体的な状況・数値を含める

### 記事構成
- 文字数：2,000〜3,000字
- 構成：書き出し → 本文（H2×3〜5） → まとめ
- 冒頭の実用情報はまとめボックス・囲み枠禁止。地の文に自然に溶け込ませる
- メタディスクリプション：120文字以内
- 内部リンク：最低2本
- 画像alt属性は必ず設定

## 制作フロー

### Phase 1: 記事テーマ受け取り
1. プリフライトチェック（`.env` の WP認証キー存在確認のみ）
2. デカ太郎が記事テーマ・キーワードを直接指定して執筆開始（ネタ帳の候補提示・ヒアリング不要）

### Phase 2: リサーチ＋構成（✅承認）
3. **デカ太郎がPerplexityで実施**：検索意図・競合構成・論点抽出・サジェストKW
4. **デカ太郎がClaudeに渡すもの**：Perplexity出力の要約（重要論点の箇条書き＋競合との差分＋勝ち筋）＋記事の目的・CTA・想定読者を1行
5. H2・H3の見出し構成案を作成し、デカ太郎に提示
6. デカ太郎が構成を承認（修正があればここで反映）

### Phase 3: 執筆＋editorial-check（✅提出）
7. 執筆（Markdown形式）
   - `01_編集部/knowledge/` 内の過去記事を参照し、文体・ふきだしのリズムを踏襲する
   - 内部リンク候補はネタ帳.mdの公開済み一覧から選ぶ
   - **分割執筆ルール**: H2が5つ以上の記事はH2単位で生成・確認を繰り返す（後半の品質劣化防止）
8. `/editorial-check` を実施（文体・構成・重複・読みやすさ）
9. 数値・断定・出典不明箇所に `FC` マークを付与してデカ太郎に提出

### Phase 4: ファクト確認＋最終監修（✅承認）
10. **デカ太郎がFC箇所のみPerplexityで確認**
11. 確認結果をClaudeに戻す → Claudeが修正反映
12. デカ太郎が最終監修（体験の嘘チェック・公開可否を判断）

### Phase 5: 公開（全自動）
13. wp-post.py で記事投稿（**必ず `--auto --publish` を付ける**）
    ```
    python3 02_技術部/スクリプト/wp-post.py "原稿ファイル.md" --auto --publish
    ```
    - `--auto`: YAMLフロントマターからtitle・slug・meta_descriptionを自動読み込み（必須）
    - `--media <ID>`: アイキャッチを別途用意した場合のみ付ける（省略可）
14. wp-post.pyの出力でslug・title・linkを確認。スラッグが`{id}-2`等の場合のみAPIで修正
15. `01_編集部/ネタ帳.md` を更新（アイデア欄から削除→公開済み欄に追記）
16. git commit + push

## 自律実行ルール

**基本原則：コスト（課金）が発生しない作業はすべて承認不要で自動実行。完了まで自律的に進行する。**

### 承認必須（これだけ）
- 外部の有料APIへの新規接続
- WordPressの設定変更（テーマ・プラグイン等）
- サイト全体に影響する一括変更
- 不可逆な本番環境の操作

### 共通制約
- 個人情報・APIキーをコードに直書きしない
- `.env`の内容をログに出力しない
- 記事は必ず /editorial-check + /fact-check を通してから投稿する

### 報告ルール
- 承認不要の作業 → 完了後に1行で事後報告
- エラー → 内容と対処を1行で報告、解決できたら続行
- 解決できないエラーのみ → 詳細を報告して指示を仰ぐ
- 作業完了時は git commit + push でGitHubを最新に保つ

## トークン節約ルール

**基本原則：最小限の入出力で最大の成果を出す。**

### 出力
- 挨拶・前置き・メタコメント・謝罪は省略。直接回答のみ
- コード修正時はdiff（変更箇所＋前後の文脈）のみ出力。ファイル全文を出力しない
- 完了報告は1行。経緯の説明は不要

### 入力（ファイル読み込み）
- 指示されたファイル、またはタスクに直接必要なファイルのみ読む
- ディレクトリの探索的スキャンはしない
- 既に読んだファイルを再読み込みしない

### セッション管理
- タスク完了後、次のタスクに移る前に `/clear` を推奨する旨を伝える
- 初回ターンでGoal・Constraints・完了条件を一括で確認し、途中の往復を最小化する
