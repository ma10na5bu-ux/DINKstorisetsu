## 編集方針

### コンセプト
30代の共働きDINKs夫婦が、時間・お金・健康のコスパを最大化する実体験を発信するメディア。DINKs暮らし・お金・仕事・趣味の4軸。

### トーンの3原則
1. 結論ファースト：前置き・時候の挨拶・「いかがでしたか？」等の感情的装飾は不要。答えを最初に出す
2. 一次情報で語る：企業サイトには書けないリアルな体験・数値・具体的な時短効果を根拠にする
3. なるべく誰も傷つかないこと：他のライフスタイルを否定しない

### NGルール
- **他のライフスタイルとの比較をしない**（子育て世帯 vs DINKs 等）
- **他人や他のものを否定しない**
- DINKsを正当化・弁明するトーンにしない
- 「DINKsの方が勝ち組」的な表現は絶対NG
- 子どもがいる人を「大変そう」と見下す表現もNG
- **医療・健康に関する記事は対象外**（獣医師に相談を案内するにとどめる）

### 記事の判断基準
「この記事を読んだ人が、CV（購入・申込・登録）行動を起こせるか？」

### コンテンツタイプ

| 集客記事 | 流入獲得 | ロングテールKW（ボリューム10〜100）を狙い、内部リンクでキラーページへ送客 |

### 文章スタイル
- 書き手（がく）の一人称視点（本文一人称は「僕」）
- はるさん（妻）・えいちゃん（柴犬）も登場
- 読者に語りかける文体（「〜ですよね」「〜ありませんか？」）
- 一次情報（リアルな体験・数値・スクリーンショット）を根拠の軸に置く
- 読者層：20〜40代の共働きDINKs夫婦
- **平易さの原則**：中学1年生でも理解できる言葉・文構造を目指す。ただし幼稚にならない。専門用語を使う場合は一言説明を添える。砕けすぎた表現（「まけてもらえる」等）は避ける。表現の適切さは監修時にフィードバックで調整する

### PREP法（全記事に適用）
すべての記事・すべてのH2に適用する。体験ストーリーの「中に」PREP構造を埋め込む形で使う。

1. **Point（結論）**：冒頭に「この記事で何がわかるか」「このH2で何が言いたいか」を1文で置く
2. **Reason（理由）**：なぜそう言えるか。データ・経験・論拠を添える
3. **Example（体験・具体例）**：がく・はるさん・えいちゃんの体験でリアルに肉付けする
4. **Point（結論）**：まとめで再強調。「だから〇〇がおすすめ」「つまり〇〇ということ」

### ベネフィット提示（全記事・特にアフィリエイト記事）
商品・サービスの機能（スペック）でなく、「それを使った後に得られる未来」を書く。

- NG: 「家計簿アプリはレシート読み取り機能がある」
- OK: 「家計簿アプリを使えば、月末の憂うつな集計作業がなくなる」

### テキスト強調ルール（太字・ハイライト）

**大原則：強調しなくても伝わる文章を書く。**
装飾が多い文章は「文章が弱い」サインになる。言葉の選び方・一文の短さ・リズムで伝える。太字・ハイライトは補助であり、主役ではない。（参考：片桐光知子「太字・赤字は逆効果？Web文章の正しい強調方法」）

**設計思想：スキャン読みでも記事のメッセージが伝わること。**
文字はほとんど読まれない。黄色ハイライトは「ここだけ読めばわかる」場所に積極的に使う。

**3種類の記法と上限：**

| 記法 | WordPress出力 | 使用場面 | 目安 |
|---|---|---|---|
| `**太字**` | `<strong>太字</strong>` | 商品名・サービス名・固有名詞のみ | 記事全体で5か所以内 |
| `<mark>テキスト</mark>` | SWELLマーカー（黄） | スキャン読みで伝えたいキーセンテンス | **記事全体で4〜6か所** |
| `<mark>**テキスト**</mark>` | SWELLマーカー（黄）＋太字 | 記事全体の核心メッセージ（1つだけ） | 記事全体で1か所まで |

**黄色ハイライトをどこに入れるか：**
- 「この記事で最も伝えたい1文」= `<mark>**太字+黄色**</mark>`（1か所）
- 「セクションの核心・読者が持ち帰るべき気づき」= `<mark>黄色のみ</mark>`（各H2に0〜1か所）
- 基準：「ここだけ読めばこのセクションの意味がわかる」文に入れる

**変換後のHTML：**
```
**太字**                   → <strong>太字</strong>
<mark>テキスト</mark>       → <span class="swl-marker mark_yellow">テキスト</span>
<mark>**テキスト**</mark>   → <span class="swl-marker mark_yellow"><strong>テキスト</strong></span>
```

**禁止パターン：**
- H2ごとに機械的にハイライトを入れる（読者が慣れてどこも目立たなくなる）
- 一文全体を太字にする（単語・短いフレーズのみ可）
- H3見出しテキストに太字・ハイライトを使う
- 体験談・ふきだし内に太字を使う（口語には不要）
- 「重要なことを言いたいから強調する」→ まず文章を削って短くする。それでもなら見出し・小見出しに昇格させる

### 視覚的読みやすさルール

**基本方針：一文を短く。箇条書きと表を積極的に使って、スマホでスキャンできる記事を作る。**

Webの文章は平均20%しか読まれないが、箇条書きは約70%が目を留める（Nielsen Norman Group, 2006）。

**一文の長さ**
- **一文40文字を目標、60文字が上限**（文章品質ルールと統一）
- **一文一義**：1つの文に盛り込む情報は1つだけ。「、」でつなぎすぎない
- 一文に情報を詰め込まない。「ポイント」と「理由・補足」は分けて書く
  - NG: `①強度やコストを自分で調整できる趣味を選ぶ。時間が読めない30代にとって、「通えない週があっても続けられる」が重要になる。`
  - OK: `強度やコストを自分で調整できる趣味を選ぶ。`（次行）`通えない週があっても続けられるかどうか、それが重要だ。`

**箇条書き**
- 「、」で区切る列挙はすべて箇条書きに変換できないか検討する（Steve Krug推奨）
- 3つ以上のポイントが並ぶときは箇条書きを優先する
- 箇条書きの目安：記事全体の20〜30%程度。やりすぎは逆効果
- 見出し・小見出しは「盛大な箇条書き」と考える。構造化こそが最強の強調

**表**
- 複数の選択肢・比較・スペックが並ぶときは表を使う
- 表は「どれを選べばいいか」を一目でわかるようにする

### タイトル
- **40文字以内**
- 感情や好奇心を刺激しつつ検索キーワードを含める
- 商品名・サービス名・地域名・具体的な状況を入れる

### 記事構成
- 文字数：2,000〜3,000字
- 構成：書き出し → 実用情報（地の文） → 本文（H2×3〜5） → まとめ
- 冒頭の実用情報はまとめボックス・囲み枠禁止。地の文に自然に溶け込ませる
- メタディスクリプション：120文字以内
- 内部リンク：最低2本
- 画像alt属性は必ず設定

### アイキャッチ画像
- 生成: generate-eyecatch.py（gemini-3-pro-image-preview → 1200×630 JPEG を自動出力）
- trim-eyecatch.py は不要（ウォーターマークなし・リサイズはスクリプト内で完結）
- 出力ファイル: `01_編集部/画像/{slug}_eyecatch.jpg`
- プロンプトファイルは `01_編集部/画像/{slug}_eyecatch_prompt.md` に保存
- `アイキャッチ画像プロンプト集.md` は読み込み不要。以下のテンプレートを使う:

```

【Scene】
{{記事の場面を英語で。夫婦30代・豆柴を自然に配置。右下15%は重要要素を置かない}}

【Art Style】
- Flat vector illustration style, NOT anime
- Minimal line work, soft rounded shapes
- Similar to modern web/app illustration (like Notion or Slack illustrations)

【Color Palette — STRICT】

【Rules】
```

## 制作フロー

### Phase 1: 記事テーマ受け取り
1. プリフライトチェック（`.env` の WP認証キー存在確認のみ。Notionチェック不要）
2. がくが記事テーマ・キーワードを直接指定して執筆開始（ネタ帳の候補提示・ヒアリング不要）

### Phase 2: リサーチ＋構成（✅承認②）
5. **並行実行**
   - **NotebookLM概要レポート（がくが手動実施）→ Perplexityでファクトチェック（がくが手動実施）**
   - **ラッコキーワードでサジェスト取得（がくが手動実施）→ Claudeで心理的ニーズ分析**
   - 渡すファイル：概要レポート・ファクトチェック結果・サジェストキーワード一覧の3点
6. **競合分析**：対策キーワードでGoogle検索上位2サイトの構成を確認（WebSearch/WebFetch）
   - 記事タイプ問わず全記事で実施
   - 競合が扱っていない独自要素（体験・データ）を特定
7. H2・H3の見出し構成案を作成し、がくに提示
8. がくが構成を承認（修正があればここで反映）

### Phase 3: 執筆＋レビュー＋監修（✅承認③）
8. 執筆（Markdown形式）
   - `01_編集部/knowledge/` 内の過去記事を参照し、文体・ふきだしのリズムを踏襲する
   - 内部リンク候補は Phase 1 で読んだネタ帳.mdの公開済み一覧から選ぶ（追加コスト0）
   - **分割執筆ルール**: H2が5つ以上の記事はH2単位で生成・確認を繰り返す（後半の品質劣化防止）
9. 並列実行: editorial-check + fact-check + アイキャッチプロンプト生成
10. がくに監修依頼（体験の嘘チェック・fact-checkの🔍人間確認項目）

### Phase 4: アイキャッチ画像（全自動）
11. generate-eyecatch.py でアイキャッチ生成・リサイズ（自動）
    ```
    python3 02_技術部/スクリプト/generate-eyecatch.py {slug}
    ```
    → `01_編集部/画像/{slug}_eyecatch.jpg` が出力される

### Phase 5: 公開（全自動）
14. 画像アップロード（curl -F multipart形式。--data-binaryはWAF 403）
15. wp-post.py で記事投稿（**必ず `--auto --publish --media <ID>` を付ける**）
    ```
    python3 02_技術部/スクリプト/wp-post.py "原稿ファイル.md" --auto --publish --media <画像ID>
    ```
    - `--auto`: YAMLフロントマターからtitle・slug・meta_descriptionを自動読み込み（必須）
    - `--media`: アイキャッチのmedia ID（`--eyecatch-id`は存在しない）
16. wp-post.pyの出力でslug・title・linkを確認。スラッグが`{id}-2`等の場合のみAPIで修正
17. `01_編集部/ネタ帳.md` を更新（アイデア欄から削除→公開済み欄に追記）
18. git commit + push

## 品質チェック

### チェック体制
- **editorial-check**: 構成・SEO（KW密度1.5〜2%）・論理整合・AI特有エラー・医療情報混入チェック（必須）
- **fact-check**: 数値データ・固有名詞・研究引用の検証（必須。editorial-checkと並列実行）
- **Geminiレビュー**: 任意。がくが「Geminiにも見せて」と指示した場合のみ実行

### 文体リファレンス
- `01_編集部/knowledge/` に高品質な過去記事5本を収録
- 執筆時に文体・ふきだし配置・構成リズムの参照元として使用

### 文章品質ルール
- **一文40文字を目標、60文字が上限**
- **一文一義**：1文に情報は1つだけ。「、」でつないで長くしない
- **見出しは「名詞＋結論」形式**：読者が見出しだけ読んでも内容がわかるようにする
  - NG: 「家計管理の方法について」
  - OK: 「家計管理｜共働き夫婦に合う3つのやり方」
- 同じ語尾が3回連続しない
- 体言止め3つ以上連続しない
- 数値の一貫性（タイトル・リード・本文・まとめで矛盾しない）
- 漢字とひらがなの配分目安：漢字20〜30%、ひらがな70〜80%（動詞・名詞は漢字、それ以外はひらがな）

### 表現NG
- 「好きです」（はるさんへの直接的愛情表現）→「感謝です」「ありがたい」
- 「必ず」「絶対」等の断言 →「〜の場合が多い」
- 症状→具体的対処指示 → 獣医師に相談を案内

## 自律実行ルール

**基本原則：コスト（課金）が発生しない作業はすべて承認不要で自動実行。完了まで自律的に進行する。**

### 承認必須（これだけ）
- 外部の有料APIへの新規接続
- WordPressの設定変更（テーマ・プラグイン等）
- サイト全体に影響する一括変更
- 不可逆な本番環境の操作

### 共通制約
- 個人情報・APIキーをコードに直書きしない
- `.env`の内容をログに出力しない
- 記事は必ず /editorial-check + /fact-check を通してから投稿する

### 報告ルール
- 承認不要の作業 → 完了後に1行で事後報告
- エラー → 内容と対処を1行で報告、解決できたら続行
- 解決できないエラーのみ → 詳細を報告して指示を仰ぐ
- 作業完了時は git commit + push でGitHubを最新に保つ

## トークン節約ルール

**基本原則：最小限の入出力で最大の成果を出す。**

### 出力
- 挨拶・前置き・メタコメント・謝罪は省略。直接回答のみ
- コード修正時はdiff（変更箇所＋前後の文脈）のみ出力。ファイル全文を出力しない
- 完了報告は1行。経緯の説明は不要

### 入力（ファイル読み込み）
- 指示されたファイル、またはタスクに直接必要なファイルのみ読む
- ディレクトリの探索的スキャンはしない
- 既に読んだファイルを再読み込みしない

### セッション管理
- タスク完了後、次のタスクに移る前に `/clear` を推奨する旨を伝える
- 初回ターンでGoal・Constraints・完了条件を一括で確認し、途中の往復を最小化する
