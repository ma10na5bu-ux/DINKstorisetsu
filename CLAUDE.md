## ✍️ 編集方針

### コンセプト
「DINKs夫婦のポジティブなラクする暮らし」を発信するメディア。

### トーンの3原則
1. ポジティブであること：DINKsの暮らしの楽しさを伝える
2. クスッと笑えること：日常のおかしみ、読んでて口角が上がる文章
3. なるべく誰も傷つかないこと：他のライフスタイルを否定しない

### NGルール
- **他のライフスタイルとの比較をしない**（子育て世帯 vs DINKs 等）
- **他人や他のものを否定しない**
- DINKsを正当化・弁明するトーンにしない（楽しさをただ共有するスタンス）
- 「DINKsの方が勝ち組」的な表現は絶対NG
- 子どもがいる人を「大変そう」と見下す表現もNG

### 公開ペース
- 初期目標：まず50本の記事を蓄積する
- カテゴリ構成は記事を書き始めてから決める

### アイキャッチ画像
- アニメチック・コミック的なイラストスタイル
- 柴犬メディア（リアルな柴犬写真）との差別化を意識
- DINKs夫婦の日常シーンをポップに描く

### コンテンツタイプと比率
| タイプ | 目標比率 | 内容 |
|---|---|---|
| 笑い記事 | 50%以上 | DINKsあるある、夫婦あるある、笑える実体験 |
| 体験記事 | 30% | 実用情報を体験ストーリーで包んだ記事 |
| 知識記事 | 20%以下 | SEO入口記事。書く場合も体験エピソードを冒頭に入れる |

### 制作フロー
1. ネタ出し：人間（まーくん）
2. 執筆：Claude
3. Geminiレビュー：自動（Gemini MCP経由）
4. 監修：人間（まーくん）— 読んで面白いか＋体験として嘘がないか＋はるさんのセリフに違和感がないかを確認
5. 監修できない領域は書かない

### マネタイズ方針
現フェーズではマネタイズは考えない。パーソナルブランドの信頼構築が先。

### 記事の判断基準
「この記事を読んだDINKs夫婦が、思わずスクショしてパートナーに送りたくなるか？」
答えがYesなら正しい記事。

### 文章スタイル
- 書き手（まーくん）の一人称視点
- はるさん（妻）も登場する
- 犬（柴犬）も登場する（ただし犬メインの記事は柴犬メディアの領域）
- 読者に語りかける文体（「〜ですよね」「〜ありませんか？」）
- ストーリーの中に情報を織り込む
- 読者層：20〜40代のDINKs夫婦・DINKsに興味がある人

### 夫婦のキャラクター
| | まーくん（夫・書き手） | はるさん（妻） |
|---|---|---|
| 性格 | 自由奔放・ポジティブ | 優しい・おだやか |
| 役割 | ボケ・突っ走る側 | ツッコミ・見守る側 |
| 記事での立ち位置 | 一人称で語る | セリフや反応で登場 |

- 夫婦の掛け合いを記事に積極的に入れる
- SWELLの「ふきだし」機能を使って会話形式で表現する
- まーくんが何かやらかす → はるさんの冷静なツッコミ、のパターンが基本

### タイトルの付け方
- 感情や好奇心を刺激しつつ、**検索キーワードを含める**
- タイトル：**40文字以内**
- 「商品名」「サービス名」「地域名」「具体的な状況」を含め、検索者が使う言葉をタイトルに入れる
- ❌「夫婦でOisix頼んでみた話」→ ✅「共働き夫婦がOisixを1ヶ月使った正直レビュー【時短効果と月額コスト公開】」
- ❌「DINKs夫婦の家計管理術」→ ✅「うちの家計簿、『趣味費』の欄だけ異常に太い【共働き夫婦の家計公開】」

### 記事構成の基本
- 文字数：2,000〜3,000字
- **冒頭に実用情報まとめを設置**（SWELLの装飾ブロック＝キャプション付きボックスで囲む。堅すぎないデザイン）
  - お出かけ記事：住所・アクセス・駐車場・料金
  - レビュー記事：商品名・価格・使用期間・一言結論
  - あるある記事：省略可（いきなり本文でOK）
- 構成：冒頭まとめ → リード文 → 本文（H2×3〜4） → まとめ
- 画像alt属性は必ず設定する

### 派生記事の考え方
- 1つの体験に複数テーマが含まれる場合、それぞれを独立記事として切り出す
- 元の日記はそのまま残し、派生記事を別途作成して相互リンクする
- 例：キャンプ日記 → 派生①テントのレビュー記事、派生②キャンプ場のレビュー記事
- **派生記事のトーン**: まーくん一人称の体験ベースで、日記よりは情報寄り。企業サイトのような三人称にしない。「友達に聞かれたから詳しく教えてあげる」くらいの距離感

### SEOルール
- メタディスクリプション：120文字以内
- H2見出し：3〜5個、各見出しにキーワードを含める
- 内部リンク：記事内に最低2本設置

## 🤖 自動化パイプライン仕様

### Phase 1：トレンド収集（毎日 9:00）
```
入力: カテゴリキーワード（DINKs, 夫婦, ライフスタイル 等）
出力: ネタ候補リスト（タイトル案×10件）
保存先: Notion「📝 ネタ帳」データベース（Notion MCP経由）
ステータス管理: 💡アイデア → ✅採用 → 📝執筆中 → 🚀公開済 / ❌不採用
```

### Phase 2：記事生成（毎日 12:00）
```
入力: ネタ帳から優先度上位1件
出力: 記事本文（Markdown）+ X投稿文×3 + Instagramキャプション×1
保存先: 01_編集部/02_執筆・原稿/YYYY-MM-DD_[記事タイトル].md
```

### Phase 3：品質チェック（生成直後）
```
Step 1: Claude自己チェック（/editorial-check）
  - 文字数（2000字以上）
  - SEOスコア（タイトル・見出し・キーワード密度）
  - トーン確認（ポジティブ・DINKs視点）

Step 2: Geminiレビュー（Gemini MCP自動実行）
  - 読んでいて不快にならないか
  - 矛盾・論理破綻がないか
  - 他のライフスタイルを否定していないか
  - 文章の自然さ・読みやすさ
  レビュープロンプト: 01_編集部/04_編集ガイドライン/Geminiレビュープロンプト.md

出力: 品質レポート（合格 / 要修正）
```

### Phase 4：公開（WordPress REST API）
```
エンドポイント: https://dinks-torisetsu.fun/wp-json/wp/v2/posts
認証: Application Password（環境変数から取得）
投稿ステータス: draft（レビュー後に手動公開 or 自動公開）
```

### Phase 5：分析・改善（毎週月曜 9:00）
```
入力: GA4データ（過去7日間）
出力: 週次レポート + 次週コンテンツ戦略提案
保存先: 03_集客部/03_アクセス解析・レポート/YYYY-WXX_週次レポート.md
```

## ✅ 承認不要（事後報告のみ・即時自動実行）

コストが発生しない以下の操作は、ユーザーの許可を待たずに即実行する：

### ファイル操作
- ローカルファイルの作成・読み込み・編集・削除・整理
- カスタムコマンド・スキルファイルの作成・更新
- CLAUDE.md・メモリファイルの更新

### コード・開発
- コード解析・静的テスト・ビルド・Lint等のコマンド実行
- Bashコマンド全般
- npm/pip等のパッケージインストール（ローカル環境）
- Git操作（commit・branch作成・ローカルでのmerge・push）
- **作業完了時は必ず git commit + push を実行し、GitHubを常に最新に保つ**

### WordPress・コンテンツ
- WordPress REST APIへの通信（記事の取得・下書き作成・更新・公開記事の更新）
- **ローカル記事の修正後 → WordPressへの自動反映**
- 記事の表現修正・構成リライト
- /editorial-check の自動実行
- 内部リンクの修正・差し替え

### エラー処理・フォールバック
- Fetch時のエラーハンドリング（リトライ・スキップ・次の候補へ移行）
- 解決可能なエラーは自己修復して続行

### 並行処理
- 複数タスクの並行実行（Agent・バックグラウンドタスク）
- 依存関係のないタスクの同時起動

## ⛔ 承認必須（コスト発生リスクがある操作のみ）

以下の操作は実行前に必ずユーザーに確認を取ること：

| 操作 | 理由 |
|---|---|
| **外部の有料APIへの新規接続** | 課金が発生する可能性 |
| **課金が発生する操作全般** | AWS・GCP・有料SaaS等 |
| **WordPressの設定変更** | テーマ・プラグイン・パーマリンク等、サイト全体への影響 |
| **サイト全体に影響する一括変更** | 全記事のカテゴリ変更・一括ステータス変更等 |
| **不可逆な本番環境の操作** | データベース変更・ドメイン設定等 |

## 📢 作業報告ルール

- **承認不要の作業** → 完了後に1行で事後報告。途中経過の報告は不要
- **エラー発生** → 内容と対処を1行で報告し、解決できたら続行
- **解決できないエラーのみ** → 詳細を報告して指示を仰ぐ
- **承認必須の操作に遭遇** → 操作内容とコスト影響を明示して確認を求める
